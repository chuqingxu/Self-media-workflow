# Self-media Workflow

这是一个面向个人自媒体、短视频制作和私域转化的工作流仓库。当前核心是：

- `skills/ai-zhuangjishi-shortvideo/`: **AI 装机师短视频全流程 skill**

它不是普通文案模板库，也不是自动发视频工具。目标是把一个个人 IP 从“想拍一个概念”，推进到“讨论成选题、写成脚本、拍成素材、剪成多平台版本、发布后复盘、再延伸成系列”。

## IP 定位

### AI 装机师

- 人设：AI 装机师
- 一句话介绍：以前帮人装电脑系统，现在帮老板装公司 AI 系统。
- 目标人群：二三线城市中小企业老板、小团队老板、传统行业老板
- 风格：老板朋友型，真人口播优先
- 平台：抖音 + 微信视频号，必要时扩展到小红书和朋友圈
- 引流：个人微信
- 承接：中转站 token / AI 使用入口，企业定制 AI 项目

核心承诺：

```text
帮二三线中小企业老板，把 AI 从“听说很厉害”装成“公司每天能用”。
```

这个 IP 不做泛 AI 博主，也不主打宏大趋势。它更像十多年前帮人装电脑系统的人：老板不需要理解所有底层技术，只需要有人帮他把入口、账号、资料、权限、流程和员工使用方式配置好。

## 真实协作边界

这个仓库特别区分两件事：

```text
真实多 agent 协作 != 一个助手扮演多个角色
```

如果只是同一个 Codex 按“策划、编剧、剪辑、审核”的视角思考，只能叫：

```text
single_agent_with_reviews
```

只有在实际调用了独立子 agent、独立线程或独立模型调用，并且留下可检查的 handoff 记录时，才能叫：

```text
real_multi_agent
```

每次交付必须标明执行模式：

```text
Execution Mode: real_multi_agent / single_agent_with_reviews / single_agent_direct
Tool Execution: none / HyperFrames / ffmpeg / transcription / browser / other
参与 agent:
各自交付:
审核结果:
```

如果没有真实子 agent 能力，就必须明说：

```text
本次为单 agent 执行，不伪装成多 agent。
```

## 唤醒词

默认唤醒词：

```text
你用ai装机师skill
```

以下写法也视为同一个触发：

```text
你用 AI 装机师 skill
你用ai装机师 skill
你用 AI装机师skill
```

当你使用这个唤醒词时，工作流不再按普通“文案助手”处理，而是默认进入真实多 agent 协同流程：

```text
Mode Gate
  -> 总控制片人判断阶段
  -> spawn 独立子 agent
  -> 子 agent 分别产出脚本、剪辑/平台方案、审核意见
  -> 总控合并
  -> 输出 Execution Mode 与审核结果
```

如果当前环境支持子 agent，就必须真实调用；如果当前环境不支持，就必须明确说：

```text
当前环境无法真实调用子 agent，本次只能 single_agent_with_reviews。
```

不能把一个助手的多视角思考说成多 agent 协作。

## 工作原理

完整链路不是“主题 -> 文案”，而是：

```text
概念
  -> 澄清问题
  -> 选题角度
  -> 平台脚本
  -> 拍摄清单
  -> 素材回收
  -> 转录与片段选择
  -> 剪映协作 / HyperFrames 自动成片
  -> 抖音 / 视频号 / 小红书平台适配
  -> 发布包装
  -> 评论区与微信承接
  -> 数据复盘
  -> 系列延伸
  -> 爆款拆解库反哺下一轮选题
```

这个 skill 会先判断你处在哪个阶段，再决定下一步，不默认一上来就写完整脚本。

## 阶段路由

| 阶段 | 典型输入 | 输出 |
| --- | --- | --- |
| 概念澄清 | 我想拍一个概念 | 追问、角度判断、目标人群、转化目标 |
| 选题打磨 | 有主题但不知道怎么讲 | 选题标题、平台角度、系列位置 |
| 脚本成型 | 要文案/口播 | 抖音版、视频号版、小红书版脚本 |
| 拍摄准备 | 要拍摄清单 | 镜头、补拍素材、口播注意事项 |
| 素材分析 | 给了视频/音频 | 转录、金句、废话段、可用片段 |
| 成片剪辑 | 要剪辑/画中画 | 剪映清单或 HyperFrames 成片方案 |
| 平台适配 | 要发多个平台 | 各平台剪辑差异、标题、封面、话题 |
| 私域承接 | 要引流/微信话术 | 评论区引导、加微信首句、分流问题 |
| 数据复盘 | 给了发布数据 | 问题归因、保留/砍掉、下一轮实验 |
| 系列延伸 | 某条反馈好 | 3-10 条延伸选题和拍摄优先级 |
| 爆款拆解 | 给了爆款链接/视频/截图，或要求找素材 | 结构拆解、借鉴点、老初版本改写、入库文件 |

## Agent 分工

真实多 agent 模式下，建议由总控 agent 调度以下子 agent：

| Agent | 负责什么 | 交付物 |
| --- | --- | --- |
| 总控制片人 | 判断阶段、拆任务、整合结果、决定下一步 | 总控 brief、最终合并报告 |
| 概念策划 Agent | 老板痛点、选题角度、系列位置 | 选题策略 |
| 口播编剧 Agent | 平台脚本、开头钩子、CTA | 抖音/视频号脚本 |
| 拍摄导演 Agent | 镜头、画面、补拍、现场执行 | 拍摄清单 |
| 素材整理 Agent | 转录、挑片段、删废话、金句标记 | 素材分析表 |
| 剪辑 Agent | 剪映清单、HyperFrames 画中画、字幕动效 | 剪辑方案或成片工程 |
| 平台优化 Agent | 抖音/视频号/小红书机制适配 | 平台版本建议 |
| 私域转化 Agent | 评论区、加微信、token/定制分流 | 私域承接话术 |
| 审核 Agent | 人设一致性、合规、转化链路、是否可拍 | 审核报告 |

最小真实多 agent 组合：

```text
总控制片人
口播编剧 Agent
剪辑/平台优化 Agent
审核 Agent
```

## 模型和 API 选择

短期建议：

```text
Codex 订阅 + Codex subagents + 本地工具链
```

这已经能覆盖概念讨论、脚本、拍摄清单、素材转录、剪辑方案、HyperFrames 成片、平台适配、审核和复盘。不要一开始就做 API 级系统，否则容易变成系统很复杂，但视频还没发几条。

后续如果要产品化，再考虑：

```text
OpenAI API / Agents SDK
视频素材存储
任务队列
自动转录
自动渲染
数据看板
```

模型分工建议：

| 任务 | 模型倾向 |
| --- | --- |
| 总控、审核、复杂剪辑决策 | 最强可用模型 |
| 选题、脚本、平台策略 | 强写作/强推理模型 |
| 转录整理、表格归档、简单改写 | 快速低成本模型 |
| HyperFrames/代码成片 | 强代码/多模态理解模型 |

## Handoff 和审计记录

真实多 agent 协作必须留下记录。推荐目录：

```text
协作记录/
  00-orchestrator-brief.md
  01-topic-strategy-handoff.md
  02-scriptwriter-output.md
  03-platform-editor-output.md
  04-conversion-review-output.md
  05-compliance-review-output.md
  06-final-merge-report.md
  audit-log.md
```

每个 handoff 文件至少包含：

```markdown
# Agent Handoff

- Execution mode:
- Agent name:
- Run/source id:
- Timestamp or execution order:
- Input received:
- Output produced:
- Decisions made:
- Assumptions:
- Risks:
- Questions for next agent:
- Files created/changed:
```

如果没有 `Run/source id` 或独立输出文件，不得把该环节称为真实子 agent。

## 审核关口

### Mode Gate

开始前声明执行模式。没有真实子 agent 能力时，禁止生成“多 agent 已协作”的措辞。

### Handoff Gate

每个真实 agent 阶段必须有独立 handoff 文件。缺文件则不能进入最终合并。

### Provenance Gate

最终内容必须能追溯到哪个 agent 产出了选题、脚本、平台改写、转化判断、合规审查。

### Contradiction Gate

合并前检查各 agent 输出是否互相冲突，例如：

- 抖音追求强冲突，但视频号要求可信
- 公开视频不能硬推 token，但私域承接可以分流
- 追求播放量，但偏离目标老板人群

### Final Claim Gate

最终报告只能按证据写：

```text
有 handoff 记录：本次使用真实多 agent 协作
无 handoff 记录：本次为单 agent 生成，并做多视角自检
```

## 剪辑执行方式

### 剪映协作模式

适合人工精修。输出：

- 素材选择
- 剪辑节奏
- 删除/保留片段
- 画中画插入点
- 字幕稿
- 封面字
- BGM/音效建议
- 平台导出规格

### HyperFrames 自动成片模式

适合实际生成视频工程和 MP4，尤其适合：

- 真人口播主画面
- AI 工具界面画中画
- 公司流程图/检查清单画中画
- 大字幕关键词强调
- 抖音竖版和视频号竖版差异版本

推荐流程：

```text
素材导入
  -> 转录
  -> 片段选择
  -> 画中画分镜
  -> HyperFrames 工程
  -> lint / inspect
  -> preview
  -> render
```

## 爆款拆解库

爆款拆解库的原则：

```text
你喂为主，我找为辅。
```

最稳定的来源是你刷到觉得有用的视频，然后把链接、截图、视频文件、标题文案或数据发给 Codex。Codex 负责拆结构、拆节奏、拆开头、拆画中画、拆转化点，并改写成“老初说 AI / AI 装机师”版本。

Codex 也可以按主题帮你找公开可查素材，但不假装能完整抓取抖音或视频号所有爆款。抖音可以参考巨量创意、第三方创意库、热点榜、公开账号；视频号内容更依赖微信内推荐流和登录状态，因此以你手动喂素材更稳。

拆解对象可以是：

```text
视频链接
本地视频文件
截图
标题 + 文案
播放/点赞/评论/转发数据
你觉得值得学的原因
```

每条拆解沉淀为一个 Markdown 文件：

```text
爆款拆解库/
  抖音/
  视频号/
  小红书/
  通用结构/
```

单条拆解会包含：

- 来源信息
- 开头 3 秒
- 核心冲突
- 口播结构
- 镜头/画中画
- 字幕节奏
- 标题封面
- 评论区引导
- 为什么可能有效
- 老初版本改写
- 系列延伸
- 承接方向

使用示例：

```text
你用ai装机师skill，拆解这个爆款视频，放进爆款拆解库。
链接：xxx
```

```text
你用ai装机师skill，我想拍“老板买了 AI 会员但员工不用”，先从爆款拆解库里找 3 个可借鉴结构，再生成脚本。
```

注意：这个库不是搬运库。只借鉴结构、节奏、镜头方法和转化逻辑，不照抄别人的口播和标题。

## 内容栏目

| 栏目 | 作用 | 示例方向 |
| --- | --- | --- |
| 装系统 | 讲公司 AI 基础能力怎么搭 | 公司用 AI 不是买会员，是先装入口、资料、流程 |
| 装软件 | 讲工具、模型、入口、token 怎么选 | 小老板先别买十几个 AI 工具 |
| 装驱动 | 讲员工、资料、权限、流程怎么接起来 | 员工不用 AI，往往不是懒，是没有接进工作 |
| 杀病毒 | 清理 AI 焦虑、伪需求、乱买软件 | 这 3 种 AI 项目，中小公司先别碰 |
| 上门案例 | 用匿名案例讲落地变化 | 一个门店老板怎么先把客服话术装进 AI |
| 老板问答 | 回答真实老板疑问 | 公司适不适合 AI？先花多少钱？员工不会用怎么办？ |

## 平台打法

### 抖音

- 30-45 秒为主
- 前 3 秒必须出现老板痛点或冲突
- 不做长铺垫，不先解释背景
- 口播密度高，字幕大，节奏快
- 结尾用评论关键词、检查清单、自测表引导

### 微信视频号

- 45-90 秒为主
- 多讲真实场景和判断逻辑
- 少一点爆点，多一点可信感
- 适合承接个人微信和后续咨询

### 小红书

- 更像经验分享和避坑笔记
- 封面像笔记，不像硬广
- 可以把口播改成清单式图文/字幕稿

## 私域承接逻辑

公开视频 CTA 不建议直接说“买 token”或“找我做项目”。更好的前端引导：

- 《中小老板 AI 装机检查清单》
- 《公司 AI 基础能力自测表》
- 《员工用 AI 不乱套的 5 个配置》

加微信后再分流：

| 用户状态 | 适合承接 |
| --- | --- |
| 只是想低成本用 AI | 中转站 token / AI 使用入口 |
| 说公司资料、客服、销售、报价、运营流程很乱 | 企业定制 AI 项目 |
| 只是好奇，还没有明确需求 | 先发检查清单，做信任沉淀 |

默认诊断问题：

```text
你现在最想让 AI 先帮公司处理哪一块，客服、销售、文档，还是员工日常提效？
```
